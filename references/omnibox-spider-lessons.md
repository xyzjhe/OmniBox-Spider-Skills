# OmniBox Spider Lessons

## 近期高价值经验

### 1. 先确认是上游挂了，还是脚本坏了

- 不要一看到分类空、搜索空、推荐空就直接改脚本；先实测上游接口状态码和返回体。
- 如果分类、推荐、搜索接口同时 500/502，优先判断为上游异常，而不是 spider 逻辑错误。
- 给用户或群友反馈时要明确说清：是“上游挂了”，还是“脚本适配失效”。

### 2. 分类/详情优先信 live DOM 与页面内状态

- 先确认页面上是否真实有内容；如果前端页面有卡片、有章节，而 spider 返回空，优先排查自己的解析逻辑。
- 对 Nuxt/SSR 站点，优先找：
  - `__NUXT_DATA__`
  - 页面内 JSON
  - 真实 `a[href*='/albums/']`
  - `ul.chapter-list > li.chapter-item`
- 能靠 live DOM + 页面状态拿到数据时，优先走这条路，通常比硬怼私有加密接口更稳。

### 3. 听书站播放链要拆层

不要默认裸播放页（如 `/playing`）就是最佳 sniff 目标；它常常只是播放器壳。

更常见的可靠链路是：
1. `detail()` 拿 `albumId + chapterIdx`
2. `play()` 调真实接口（如 `/api/play_token`）拿直链
3. 拿不到再 fallback 到带章节上下文的页面（如 `/audios/{albumId}/{chapterIdx}`）去 sniff

对“集合壳页面”类站点，`detail()` 应先拿真实详情页/真实章节上下文，`play()` 再取直链，不能混成一层。

### 4. 页面真实播放页已知时，优先交给 SDK 嗅探

- 已知真实播放页 URL 时，优先 `OmniBox.sniffVideo(realPlayPageUrl)`，比自己手撸 Playwright 点击链更轻更稳。
- 但要注意：`sniffVideo` 也可能受运行环境 bug 影响，不能盲目把它当唯一方案；能直解接口就优先直解。

### 5. 遇到加密 API，先去前端 bundle 抄真逻辑

先从页面引用的 `_nuxt/*.js` 或前端 bundle 里搜：
- `payloadKey`
- `payloadVersion`
- `encryptRequest`
- `decryptResponse`
- `xchacha`
- `poly1305`

真实站点逻辑可能是：
- 请求 v1、响应 v2
- 响应体需要特殊处理，例如先 `reverse(cipher)` 再解

先抄前端真逻辑，再把 spider 里的解包对齐，效率远高于盲试 nonce/AAD 切片。

### 6. tingyou.fm 这次确认的关键点

- 分类与详情可以稳定依赖页面结构和 `__NUXT_DATA__`
- 播放主链应走 `/api/play_token`
- `/audios/{albumId}/{chapterIdx}` 更适合做 sniff fallback，而不是裸 `/playing`
- `play_token` 的 v2 响应需要按前端真实逻辑解包：
  - `nonce = bytes[1..24]`
  - `cipher = bytes[25..]`
  - 当 `version === 2` 时，先 `reverse(cipher)` 再解密

### 7. 交付默认最小补丁

- 用户或群友贴来原始脚本时，先固化原版基线，再做最小补丁。
- 不要随意改域名、头注释、依赖、站点配置块、线路顺序，除非问题本身要求。
- 验收前要核对 diff，防止“嘴上说最小改动，实际变成重写稿”。

### 8. 交付顺序：先文件，后说明

- 在群里交付爬虫修复时，优先直接发文件；说明尽量短，别长篇铺垫。
- 文件与说明不要分散太开：要么同一条一起发，要么先发文件再回复说明。

### 9. 日志策略：核心常驻，诊断可开关

常驻日志保留：
- `detail.out`
- `play.req`
- `play.out source=api/page`
- 关键错误

高频诊断日志如：
- `api.raw`
- `api.payload.meta`
- `decrypt.variant.fail`

应做成环境变量开关（例如 `TINGYOU_VERBOSE_API=1`），避免正常使用时刷屏。

### 10. 验收别只看语法

- `node --check` 只能验语法，不能保证运行时逻辑对。
- 关键链路仍要实测：分类、详情、播放至少各走一遍。

### 11. 吸收 drpy-node 技能时，优先学“流程分诊”，不要误学成“宿主混用”

- drpy-node 技能里最有价值的，不是把 `var rule = {}` 或它的全局函数硬搬进 OmniBox，而是它把任务拆成：建源、普通结构问题、播放专项、仓库交付 这几层。
- 以后遇到“写源 / 修源”请求，先判断当前卡在哪一层，再动手；不要一上来把建源、lazy、上传、标签整理一起做成大杂烩。
- 如果目标本来就是 OmniBox spider，就继续遵循 OmniBox 的五方法与 SDK；如果目标明确是 ds / drpy-node，再切到 `var rule = {}` 的那套输出。

### 12. 新建源先交最小可用，不要第一版就贪全

- 第一阶段先把站点类型判断清楚：DOM 规则更稳，还是 API 更稳。
- 再按 `home/category/search/detail/play` 或 ds 对应的 `推荐/一级/二级/搜索/lazy` 分阶段推进。
- 第一版优先确保：能进列表、能进详情、能拿到一条真实可验证的播放链；增强项（标签、美化、额外线路、仓库上传）放后面。

### 13. 播放修复要防“假通过”

- 不要因为返回了一个页面 URL、`parse:1`、播放器壳页、`play.html` 中转页，就直接认定播放已修好。
- 至少要继续确认：
  - 这是不是空壳播放器页
  - 客户端是否真的能播，而不是只有浏览器前端勉强可见
  - 是真实直链 / 可嗅探页，还是只是站内中转页
- 如果只能先交 fallback，也要明确写成“临时兜底”而不是“播放已完全修复”。

### 14. 下载区 / 网盘分享型站点，优先按“分享 = 线路，文件 = 剧集”建模

- 如果详情页主展示的是下载按钮、复制链接、115/百度/夸克/阿里分享地址，这类站点默认优先按网盘源建模，而不是普通在线播放源。
- 正确优先级通常是：
  1. 每个分享链接作为一条线路
  2. 再用 SDK 展开这个分享下的具体视频文件 / 剧集
  3. `play()` 再用 `shareURL + fileId` 去取真实播放信息
- 不要把“分享链接按钮本身”直接当成最终 episode，也不要直接把下载链接当最终播放直链。

### 15. 线路数量与线路命名都要克制，别制造伪差异

- 多线路（如 `服务端代理 / 直连 / 本地代理`）应基于仓库现有配置或明确需求，不要默认所有网盘都自动展开多线路。
- 如果官网下载区只有 1 条记录，而脚本解析出了 2 条线路，优先怀疑是选择器重复命中、同一条分享被解析两次，而不是网站真的有两条。
- 对同类网盘多分享，线路名要做稳定区分（如 `115网盘1 / 115网盘2`）；但如果只有 1 条真实分享，不要凭空加编号或重复生成线路。

### 16. 网盘显示名优先信 URL host，谨慎使用 SDK displayName

- 已知 host 很明确时（例如 `115cdn.com / 115.com`），显示名优先信 URL 自身，不要盲信 SDK 的 `displayName`。
- 否则可能出现：明明是 115 分享，却被 SDK 误报成 `UC网盘`，导致前端线路名全错。

### 17. 分类筛选前先分清“子分类入口”还是“class 筛选”

- 很多 MAC 变体站里，像 `韩剧 / 日剧 / 泰剧 / 欧美剧 / 纪录片 / 儿童` 这种一排，其实是独立 `type_id` / 子分类入口，不是 `class=韩剧`。
- 真正的 `class` 筛选通常是另一排：如 `古装 / 战争 / 喜剧 / 犯罪 / 剧情 / 历史`。
- 在补筛选前，应先实测真实 `vod/show/...` 路由，再决定某个筛选项该映射到：
  - `type_id / subType`
  - `class`
  - `area`
  - `lang`
  - `year`
  - `letter`
  - `by`

### 18. 详情页若先返回 JS validator，不要把那层 HTML 当真实详情

- 如果详情页先返回形如：
  - `document.cookie = "fl_js_validator_...";`
  - `window.location.reload();`
  这类页面，说明当前请求先被前端验证层拦住了。
- 这时不要继续拿这 200 响应正文去解析播放列表；应优先补 cookie 后重试，再决定是否回退 API。
- 否则很容易出现：明明是验证页，却被脚本当成“没有线路的详情页”继续往下走。
